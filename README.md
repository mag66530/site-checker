# Site Checker

Автоматическая проверка доступности страниц СМУ, ИМП, МПЭ.

## Вкладки

- **Чек-лист 15 мин** - ежедневная проверка контент-менеджером (профили
  Быстрая / Стандартная / Полная, 404 из Метрики, Telegram-уведомления).
- **Чек-лист 30 мин** - еженедельная проверка помощником/джуном: случайная
  выборка 300-500 URL без повторов (ротация 30 дней) + ручные пункты
  (почта, вебмастер/GSC, метрика, подмена номера) с галочками.
- **Проверка форм** - реальная отправка форм на боевых сайтах через Playwright
  (настоящий Chromium): заполнение, отправка, проверка успеха + больше 20
  UX- и безопасность-пунктов по каждой форме. Что и как именно проверяется -
  раздел «Проверка форм» ниже.
- **Скорость страниц** - оценка скорости через Google PageSpeed Insights
  (Lighthouse) по типам страниц, сравнение с прошлым периодом и конкретные
  рекомендации (какие файлы и на сколько тормозят). Отчёт - Excel. Раздел
  «Скорость страниц (PageSpeed)» ниже.

## Что проверяется

- 🏠 Главные страницы поддоменов
- 📁 Страница /catalog/
- 📂 Категории каталога (листинги и разделы-витрины - по-разному)
- 🏷️ Фильтры и теги
- 🛒 Карточки товаров (из базы листингов, fallback - sitemap.xml)
- 🧩 Структура страниц: цена (в рублях / по запросу), кнопки заказа,
  H1, хлебные крошки, шапка/подвал, плитка тегов «Часто ищут», блок
  «Похожие товары» (карусель, цены, перелинковка), отзывы с проверяемым
  источником (Flamp/2ГИС/карты - подпись сервиса без ссылки на него),
  сортировка листинга…
  Что именно и КАК проверяет каждый столбец - описано в комментариях к
  заголовкам листа «Структура страниц» в xlsx-отчёте (навести мышь).
- 🔤 Битые переменные в текстах ({{city}}, %price%, #MIN_PRICE#, undefined, [object Object])

## Проверка форм

Вкладка «Проверка форм» реально заполняет и отправляет формы на боевых
сайтах через Playwright (настоящий Chromium), не эмулирует запросы -
поэтому видит и JS-обработчики сайта, и его реальные ответы. По каждой
форме пишется строка (иногда несколько) в лист «Логи» файла
`log_forms.xlsx`. У части столбцов лога заголовок несёт комментарий с
механикой проверки (наведите мышь) - по образцу листа «Структура
страниц». Ниже - как устроены самые важные пункты:

- **Подсказки полей понятны** - placeholder/подсказка поля не должна
  ПРОТИВОРЕЧИТЬ его назначению (ловим copy-paste-баги вроде поля
  «Телефон» с подсказкой про e-mail). Назначение поля определяется из
  type/autocomplete/name/подписи, а не из самого placeholder - затем
  сравнивается с тем, на что похож текст подсказки.
- **Ошибки валидации отображаются корректно** - проверяется не сам факт
  наличия правил валидации, а появляется ли видимое сообщение при
  пустом/невалидном вводе. Нативную HTML5-валидацию ловим безопасно
  через `checkValidity()` без реальной отправки; кастомную (свой JS
  сайта) - контролируемым пустым сабмитом под двойной защитой (перехват
  submit + обрыв запроса) - заявка никогда не уходит.
- **Форма корректно на мобильных устройствах** - страница открывается
  свежим мобильным контекстом (390px, touch) и измеряется: нет ли
  горизонтального скролла, не мельче ли кнопки/поля ~44px (минимальная
  тач-цель). Отдельно, на трёх ширинах экрана (телефон/планшет/десктоп),
  проверяется, что элементы формы не налезают друг на друга и форма
  остаётся пригодной (видны и поле, и кнопка). Пишет отдельные строки в
  «Логи» с названием «Мобильная вёрстка: …».
- **Форма защищена от CSRF** - пассивно, без единого лишнего запроса:
  в уже заполненной форме ищем скрытое `<input type="hidden">` с именем
  вроде csrf/sessid/_token/authenticity_token/xsrf/nonce.
- **Данные формы корректно передаются на сервер (видно в DevTools →
  Network)** - пассивно, без единого лишнего запроса, на той же самой
  легитимной отправке (включая заказы): тул снимает значения полей ПРЯМО
  ПЕРЕД кликом отправки и слушает реальный сетевой POST-запрос той же
  отправки, сверяя, что каждое заполненное значение долетело до сервера
  (устойчиво к переформатированию телефонной маски и обрезке длинного текста
  по maxlength сайта - это не считается потерей данных). Ловит баг «сайт
  нарисовал Спасибо, но реально отправил не то, что было в полях». **Не
  путать со следующим пунктом** - тот проверяет обратное направление: что
  БИТЫЕ данные сервер отклонит, а не что ХОРОШИЕ не потеряются по дороге.
- **Проверка данных на сервере (нельзя обойти через DevTools)** -
  только под галочкой «Проба серверной валидации»: тул через
  `page.evaluate()` снимает html5-ограничения у поля (required/pattern/
  maxlength/type) и жмёт настоящую кнопку отправки - имитация того, что
  человек делает руками в консоли браузера. До 3 вариантов: пустое
  обязательное поле, некорректный e-mail, слишком длинный текст. Если
  сервер принял хоть один - заявка реально ушла в админку клиента (с
  тестовой меткой в имени, чтобы её можно было отличить и удалить).
- **Ограничено количество запросов (защита от спама и ботов)** -
  пассивный слой проверяется ВСЕГДА: капча-виджет по классу/скрипту
  (reCAPTCHA/hCaptcha/Cloudflare Turnstile/Яндекс SmartCaptcha) и
  honeypot-поле (скрытый input с «ловушечным» именем вроде hideit/
  honeypot/trap). Активный слой - только под отдельной галочкой
  «Активная проверка лимита запросов» (по умолчанию выключена, явное
  предупреждение в подсказке к галочке): 3 быстрые повторные отправки
  подряд валидными тестовыми данными, проверяем, блокирует ли сайт
  позднюю попытку.
- **Модальные окна работают корректно** - «открывается»: клик по
  кнопке-триггеру и проверка, что внутри появившегося контейнера реально
  есть поле ввода (не пустой fallback-селектор). «Закрывается»: по
  очереди пробуются три штатных способа - крестик/кнопка закрытия, Esc,
  клик по фону вне окна модалки - и фиксируется, какой сработал первым;
  если не сработал ни один - честное «Нет», без «проверить вручную».

Проверки с побочными эффектами (серверная валидация, активный залп
лимита, проба файлов, XSS) всегда идут под отдельными галочками,
выключенными по умолчанию - обычный прогон форм на боевых сайтах их не
трогает.

## Скорость страниц (PageSpeed)

Вкладка «Скорость страниц» проверяет скорость через Google **PageSpeed Insights
API** (Lighthouse) - тот же движок, что на pagespeed.web.dev, только массово и с
историей.

**Что делает:**

- гоняет выборку URL проекта через PageSpeed (**desktop + mobile**, `locale=ru`);
- собирает оценку производительности (0-100) и метрики Lighthouse
  (FCP, LCP, CLS, TBT), а также «полевые» данные CrUX, если Google их отдал;
- считает средние по типам страниц (Главная / Каталог / Категория / Товар /
  Фильтр) и **Δ к прошлому периоду** (прошлый прогон / ~неделя / ~месяц);
- собирает **конкретные рекомендации**: не просто «оптимизируйте картинки», а
  какие именно файлы и на сколько (КБ/сек) - прямо из `details` Lighthouse;
- пишет **Excel-отчёт** (листы «Сводка по типам», «Детально», «Рекомендации»).

**Ключ PageSpeed API** (обязателен - без него Google даёт жёсткий лимит):

1. [Google Cloud Console](https://console.cloud.google.com/) → создать/выбрать проект;
2. включить **PageSpeed Insights API**;
3. Credentials → Create credentials → **API key**;
4. положить в секреты приложения:
   - `pagespeed_api_key` - общий для всех проектов, или
   - `pagespeed_api_key_<проект>` (напр. `pagespeed_api_key_smu`) - на проект.

   Локально - в `.streamlit/secrets.toml` или переменной окружения
   `PAGESPEED_API_KEY`. Ключ **никогда не коммитится**.

**История и сравнение периодов** хранятся локально в `pagespeed_data/{проект}.csv`
(в git не попадает). На Streamlit Cloud диск эфемерный - чтобы не терять историю,
на вкладке есть кнопки **⬇ Скачать историю / ⬆ Загрузить историю** (CSV). Для
регулярного накопления удобнее локальный запуск или CLI:

```bash
# выборка по 5 страниц каждого типа, сравнение с прошлым прогоном
PAGESPEED_API_KEY=AIza... python pagespeed_run.py --project smu --per-type 5 --compare prev
# свой список URL
PAGESPEED_API_KEY=AIza... python pagespeed_run.py --project smu --scope list --urls-file urls.txt
```

**Ограничения PageSpeed** (не бага скрипта): проверка медленная (каждый URL ~
10-40 с × desktop+mobile), поэтому берём выборку, а не все 300-500 URL; оценка
лабораторная (один прогон, возможен разброс в несколько пунктов); если сайт
блокирует зарубежные IP - серверы Google могут не открыть страницу и вернут
ошибку.

## Запуск локально

```bash
pip install -r requirements.txt
streamlit run app.py
```

Откроется в браузере на http://localhost:8501

## Деплой на Streamlit Cloud

1. Залить этот репозиторий на GitHub
2. Зайти в https://streamlit.io/cloud
3. New app → выбрать репозиторий → Main file: `app.py`
4. Deploy

Секреты (Manage app → Settings → Secrets):
- `pagespeed_api_key` или `pagespeed_api_key_<проект>` — ключ PageSpeed для
  вкладки «Скорость страниц».
- `gsc_service_account_<проект>` — base64 JSON сервисного аккаунта Google для
  источника «Google» в «404 в индексе» (Search Console API, работает на облаке
  без браузера). Сервисный аккаунт должен быть добавлен пользователем в
  GSC-ресурс проекта (Search Console → Settings → Users). Ресурс задаётся в
  `projects/<проект>.json` (`gsc_resource`), по умолчанию `sc-domain:<домен>`.

## Обновление каталогов

Каталоги хранятся в виде CSV-файлов в папке `catalogs/`:

- `{proj}-subdomains.csv` - поддомены (колонки: url, city)
- `{proj}-catalog.csv` - каталог категорий и фильтров (колонки: url, type)
- `{proj}-categories.csv` - отдельный файл актуальных категорий (только для СМУ)
- `{proj}-products.csv` + `{proj}-products-meta.json` - база товарных ссылок,
  собранных с листингов (см. ниже)

Чтобы обновить - конвертировать новый xlsx в CSV через `convert_catalogs.py`
(нужны исходные xlsx из старого Node.js проекта в папке `/home/claude/site-checker/catalogs/`).

## База товаров с листингов (обновлять раз в месяц)

Товары для проверки берутся из базы, собранной с реальных страниц категорий
(первая страница листинга каждой категории, без пагинации). На Streamlit
Cloud месячный кэш на диске не живёт, поэтому база лежит в git:

```bash
python collect_products.py smu --proxy http://login:pass@host:port
python collect_products.py imp
python collect_products.py mpe
git add catalogs/ && git commit -m "Обновление базы товаров" && git push
```

Количество товаров и дата сбора видны в карточке «Каталог проекта».
Через 30 дней база помечается устаревшей (⚠) - пора пересобрать.
Кнопка «♻ Сбросить кэш товаров» чистит локальный кэш приложения
(sitemap + каталог); сама база в репозитории при этом не трогается.
Если базы нет - товары берутся из sitemap.xml, как раньше.

## Профили проверок

| Профиль | Городов | Категорий | Фильтров | Товаров | Время |
|---|---|---|---|---|---|
| Быстрая | 3 (Москва + 2) | 3 | 3 | 2 | ~1-2 мин |
| Стандартная | 6 (Москва + 5) | 5 | 5 | 3 | ~3-5 мин |
| Полная | 11 (Москва + 10) | 10 | 10 | 5 | ~10-20 мин |
| Свои настройки | вручную | вручную | вручную | вручную | - |

## Ротация выборки

Скрипт помнит какие URL проверялись раньше. При формировании случайной
выборки им даётся меньший вес (0.3 от обычного), чтобы постепенно
покрывался весь каталог. Окно: 7 дней для ежедневного чек-листа,
30 дней для еженедельного. История хранится в `cache/history-{proj}.json`.

## Прокси (для проектов с блокировкой по IP)

Если сайт проекта блокирует зарубежные IP (например Streamlit Cloud в США),
можно настроить прокси. В JSON-конфиге проекта (`projects/{id}.json`) ставим
флаг `"use_proxy": true`.

Сами креды прокси хранятся в Streamlit Secrets (Settings → Secrets):

```toml
proxy_url = "http://login:password@host:port"
```

Конкретный пример настроек:
- СМУ (`stalmetural.ru`) - `use_proxy: true` (блокирует американский IP)
- ИМП (`inmetprom.ru`) - `use_proxy: false`
- МПЭ (`mepen.ru`) - `use_proxy: false`
- Custom-режим - прокси не используется никогда

Если в будущем какой-то проект начнёт блокировать - просто меняем `use_proxy: true`
в его JSON-конфиге.
